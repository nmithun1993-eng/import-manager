Assets

Bulk-import assets with full hierarchy. Site → Building → Floor → Space → Subspace — each level is validated against its parent so duplicate-name conflicts (e.g. two "Ground Floor" spaces in different buildings) get caught at validation time.

How hierarchy validation works
For each row you fill in as much hierarchy as you know. The deepest level provided becomes the asset's location.

Examples:
Site = HQ alone → asset attached to site
Site = HQ, Building = Tower A → asset attached to building
Site = HQ, Building = Tower A, Floor = Ground, Space = Lobby → asset attached to space
… + Subspace = Reception → asset attached to the sub-space

The validator looks up each name in the context of its parent: a Building "Tower A" must belong to Site "HQ", a Floor "Ground" must belong to Building "Tower A", and so on. If the chain doesn't match, the row fails with a clear message. Numeric IDs work in any cell as a shortcut.

Job Plans

One row = one task. Rows that share the same Sl No are merged into a single Job Plan; within that, rows that share the same Section Name become one section. Change Sl No to start a different Job Plan.

How grouping works (read this first)
The importer groups rows by Sl No:
  • All rows with Sl No = 1 become one Job Plan. All rows with Sl No = 2 become a different Job Plan, and so on.
  • The first row of each Sl No decides Job Plan Name, Scope, Description and Publish for the whole job plan.
  • Inside each Job Plan, rows that share the same Section Name become one Section. Leave Section Name blank and tasks land in "Section 1".
  • Each row's task fields (Name, Criticality, Input Type, Options, Photo, etc.) become one task inside its section.
Input Type: leave blank for a no-reading task; Text for a free-text answer; Numeric for a number (use Deviation like 10,90 to flag out-of-range readings); Option for a pick list — fill Options with comma-separated values, e.g. OK, Needs Repair, Replace.

Photo? — set to Yes to make a photo attachment mandatory for that task.

Skill Set — must match an existing Skill name in your account (lookup is best-effort; unknown skills are silently skipped).

If tasks don't show up in Facilio after import — click Inspect schema. It scans your existing job plans, finds one with tasks, and logs the exact field names the API uses, so we can match them.

Job Plans — V2 (two-sheet workflow)

Fill Sheet 1 — Job Plans with the basics (Name, Scope, Description). Then switch to Sheet 2 — Tasks and add tasks, linking each task to a plan by its Job Plan Name. When you click Import, both sheets are validated and merged: one job plan per row on Sheet 1, with all matching tasks from Sheet 2 attached.

How V2 differs from V1
V1 uses a single sheet — one row per task, with the job plan name repeated. Easy to import, but less intuitive for non-technical users.

V2 splits the same data across two sheets:
  • Sheet 1 — Job Plans: one row per job plan. Define Name, Scope, Description, Publish flag.
  • Sheet 2 — Tasks: one row per task. Reference the parent plan by its Job Plan Name. Section Name groups related tasks within a plan.
On import, V2 looks up each task's Job Plan Name in Sheet 1, groups them, and creates one job plan per Sheet 1 row using the same proven backend as V1. Job Plan Name is the unique identifier — it must match exactly between sheets (case-insensitive).

Record History

Enter a Record ID to see a merged timeline of workflow logs, script logs and audit logs for that record — so you don't have to flip between three screens in Facilio.

What's fetched, and how filtering works
For the given Record ID, we hit three endpoints in parallel:
  • Workflow Log — api/v3/modules/workflowLog
  • Script Log — api/v3/modules/workflowRuleLogs
  • Audit Log — api/v3/modules/auditLogs
Each entry is matched to the Record ID by checking common foreign-key fields (parentId, recordId, ticketId, parent.id, moduleRecord.id, record.id) as well as a deep JSON scan for the ID — so it picks up references regardless of which field the log stores them in.

If you don't see logs you expect, increase "Rows per source" — we pull the most recent N entries from each log and filter client-side. Default 200 is fine for most cases.

Scoring Rules

Attach scoring rules to questions on an existing template. One grid row = one condition. Rows are grouped by Template ID → Question ID at import time and sent as a single {rules:[...]} POST per template.

Load scorable questions from a template

Paste one or more Template IDs (comma-separated) and click Load. We'll fetch the templates, walk their pages and questions, and pre-populate the grid with only the question types that support scoring (Single/Multi Select, Number, Decimal, Date/Time). For option-type questions, one row per option is pre-filled with the option's ID and label — you just fill in the Score.

Operator codes: 9 = less than, 11 = greater than, 20 = on date. Pull exact codes from a DevTools capture if you need other operators.
